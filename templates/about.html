System Architecture Reference

The Priority Inversion Problem

In real-time operating systems, priority inversion is a critical failure state where a high-priority task is indirectly preempted by a medium-priority task because they both depend on a shared resource currently locked by a low-priority task.

In the context of the Engine: If the Thruster Array (High Priority) needs power data (Low Priority) to recalculate trajectory, but Life Support (Medium Priority) preempts the power calculation, the Thrusters stall. This leads to catastrophic failure in time-critical environments like the Mars Pathfinder.

Implemented Countermeasures

  • Standard Scheduling (No Protocol)
    No active mitigation. Medium-priority tasks have free rein to preempt the resource-holding low-priority task, causing the high-priority task to wait indefinitely.

  • Priority Inheritance Protocol (PIP)
    When a high-priority task blocks on a resource, the task holding the resource temporarily "inherits" the high priority. This prevents medium-priority tasks from preempting it. Once the resource is released, the priority drops back to normal.

  • Priority Ceiling Protocol (PCP)
    Every resource is assigned a "priority ceiling," which is the highest priority of any task that may lock it. When a task locks a resource, the system dynamically ensures it won't be preempted inappropriately, completely preventing both deadlocks and chained blocking.
